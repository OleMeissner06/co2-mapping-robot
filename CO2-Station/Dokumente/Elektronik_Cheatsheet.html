Die 4 Kabel – Was steckt wohin?

Der Pimoroni SCD41 Breakout (PIM587) braucht genau 4 Verbindungen zum Raspberry Pi, hergestellt über die bestellten Female-Female Jumper-Kabel.

VCC
Pin 1 · 3.3V · Stromversorgung für den Sensor
GND
Pin 6 · Masse · Minus-Pol, schließt den Stromkreis
SDA
Pin 3 · Serial Data · Hier fließen die Messwerte (CO₂, Temp, Hum)
SCL
Pin 5 · Serial Clock · Takt – gibt den Übertragungsrhythmus vor
Achtung: Immer auf 3.3V (Pin 1) stecken, nicht auf 5V (Pin 2/4). Der SCD-41 verträgt keine 5V und kann dabei kaputtgehen. Im Zweifel: pinout.xyz aufrufen.

Was ist ein Takt?

Ein Takt ist ein Signal das regelmäßig zwischen 0 und 1 wechselt – wie ein Metronom. Jedes Mal wenn es auf 1 springt, darf ein Bit übertragen werden.

SCL – Takt (Metronom)
SDA – Daten (synchron zum Takt)

Der Pi gibt den Takt vor (Master), der SCD-41 richtet sich danach (Slave). Ohne Takt würden Bits unkontrolliert ankommen – wie ein Gespräch wo beide gleichzeitig reden.

I2C – Das Kommunikationsprotokoll

I2C (gesprochen: "I-squared-C") ist die vereinbarte Sprache zwischen Pi und Sensor. Besonderheit: mehrere Geräte können an denselben zwei Kabeln hängen – jedes hat eine eindeutige Adresse.

In Python wird der Sensor über seine Adresse gefunden – automatisch, du musst nichts eintippen:

i2c = board.I2C()
sensor = adafruit_scd4x.SCD4X(i2c)   # sucht automatisch nach 0x62

Hexadezimal – Warum 0x62?

Das 0x vor einer Zahl bedeutet: diese Zahl ist hexadezimal (16er-System, nicht 10er). In der Elektronik ist das der Standard für Adressen und technische Werte.

SystemBasisZiffernBeispiel
Dezimal100–998
Hexadezimal160–9, A–F0x62
Binär20, 101100010

0x62 umrechnen: 6 × 16 + 2 = 98 (Dezimal) – das ist die Hausnummer des SCD-41 auf dem I2C-Bus.

Siehst du irgendwo im Code 0x..., ist das immer eine Adresse oder ein technischer Festwert – kein Rechenfehler.

GPIO – Die 40 Pins am Raspberry Pi

GPIO steht für General Purpose Input/Output. Die meisten Pins sind frei programmierbar – du entscheidest per Python was sie tun. Einige sind aber fest belegt.

TypBeispiel-PinsAufgabe
3.3VPin 1Feste Stromversorgung – immer an
5VPin 2, 4Feste Stromversorgung – immer an
GNDPin 6, 9, ...Masse – immer verfügbar
GPIOPin 7, 11, ...Frei programmierbar
I2C SDAPin 3Fest für I2C reserviert
I2C SCLPin 5Fest für I2C reserviert
Vollständige Pinbelegung immer auf pinout.xyz nachschlagen – besonders bevor du einen neuen Sensor anschließt.

Digitale Signale – HIGH & LOW

Der Pi kommuniziert nicht mit Buchstaben oder Zahlen direkt – er kennt nur zwei Zustände auf jedem Pin:

ZustandSpannungBedeutung
HIGH~3.3VBit = 1
LOW~0VBit = 0

SDA wechselt im Rhythmus des Takts zwischen HIGH und LOW. Jeder Wechsel ist ein Bit. Erst 8 Bits zusammen (= 1 Byte) ergeben eine lesbare Zahl.

SDA – Beispiel: Byte 0x62 (Adresse des SCD41)

Logikspannung – Warum Kompatibilität wichtig ist

Jedes Gerät hat eine Logikspannung – die Spannung die es als HIGH und LOW versteht. Stimmen die Spannungen nicht überein, werden Daten falsch interpretiert oder der Sensor wird beschädigt.

GerätLogikspannungKompatibel mit SCD41?
Raspberry Pi 53.3V✓ Ja
Pimoroni SCD413.3V✓ Ja
Arduino Uno5V✗ Nein – Level Shifter nötig
Regel: Vor jedem neuen Bauteil im Datenblatt prüfen welche Logikspannung es erwartet. 5V auf einem 3.3V-Pin kann den Sensor dauerhaft zerstören.

Was passiert wenn sensor.CO2 aufgerufen wird?

Der SCD41 sendet pro Messung 9 Bytes über SDA: CO₂ (2 Bytes + CRC), Temperatur (2 + CRC), Luftfeuchtigkeit (2 + CRC).

CRC (Cyclic Redundancy Check) ist eine Prüfsumme. Der Sensor berechnet sie aus den Datenbytes und schickt sie mit. Der Pi berechnet sie nochmal aus den empfangenen Bytes und vergleicht. Stimmt es nicht → Messung wird verworfen, kein Absturz.

# Das siehst du in deinem Code später:
print(sensor.CO2)              # z.B. 1024   (ppm)
print(sensor.temperature)      # z.B. 21.4   (°C)
print(sensor.relative_humidity) # z.B. 52.1  (%)

systemd – Autostart unter Linux

systemd ist der Dienst-Manager unter Linux. Er startet beim Booten alle Hintergrunddienste – SSH, WLAN, und nach unserer Konfiguration auch logger.py und dashboard.py. Ein Dienst wird als einfache Textdatei (.service) definiert.

Aufbau einer Service-Datei:

[Unit]
Description=CO2 Logger          # Name des Dienstes
After=local-fs.target           # Erst starten wenn Dateisystem bereit ist

[Service]
ExecStart=/usr/bin/python3 /home/ole/co2-mapping-robot/CO2-Station/Skripte/logger.py
WorkingDirectory=/home/ole/co2-mapping-robot/CO2-Station/Skripte/
User=ole                        # Als Nutzer ole ausführen, nicht als root
Restart=on-failure              # Neustart bei Absturz

[Install]
WantedBy=multi-user.target      # Bei normalem Boot aktivieren
Wichtig: systemd kennt keine PATH-Variable. Deshalb immer den absoluten Pfad zu Python angeben (/usr/bin/python3), nicht nur python3.

Abhängigkeit zwischen Diensten – das Dashboard braucht den Logger:

[Unit]
After=local-fs.target co2-logger.service   # Warten bis Logger läuft
Wants=co2-logger.service                   # Logger als Abhängigkeit markieren

Die wichtigsten Befehle:

BefehlBedeutung
sudo systemctl enable co2-logger.serviceAutostart beim Booten aktivieren
sudo systemctl start co2-logger.serviceDienst jetzt sofort starten
sudo systemctl stop co2-logger.serviceDienst stoppen (bleibt enabled)
sudo systemctl status co2-logger.serviceStatus & Logs anzeigen

Service-Dateien liegen unter /etc/systemd/system/ und werden mit sudo nano erstellt. Das Projekt nutzt zwei: co2-logger.service und co2-dashboard.service.
